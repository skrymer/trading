Every piece of new work starts on a **feature branch**. Never `git commit` + `git push` on `main` directly.

**Exception — deploy version bumps:** The version-only changes to `udgaard/build.gradle` / `midgaard/build.gradle` made by `deploy-prd.fish` ARE fine to commit directly to `main` (commit style: `Deploy version bump: midgaard 1.0.58, udgaard 1.0.54`). The user confirmed this 2026-05-21. These are mechanical, generated by the deploy script, and carry no logic to review — the branch→PR ceremony adds nothing.

**Why:** During the ovtlyr signal-storage session, the main feature went through a proper PR (#26), but the follow-up *code* fixes afterward (HTTP-status logging, base-URL config, resumable backfill, "Invalid Stock" handling) were committed straight to `main` — each one when the user said "commit it" / "handle it". They never got the PR review step. The user called this out and wants it to never happen again: all real work goes branch → PR → merge. Version bumps are the one carve-out.

**How to apply:** Before making the first code change of any task, create a feature branch (`git checkout -b <type>/<short-name>`). Commit there, push the branch, open a PR. This holds even for small fixes and even when the user says "commit it" / "handle it" mid-flow — interpret that as "commit to the feature branch", not "commit to main". If no feature branch exists when a change is requested, create one first. The sole carve-out is deploy version bumps (see Exception above) — commit those straight to `main`. Pushing/PR-ing still needs explicit user permission per [[feedback-never-push-without-permission]]; this rule is about *where* commits land, not *when* to push.
