# Same-day cohort ranking is a first-class `StockRanker` capability, wired only at the co-resident selection site

To support a ranker that **blends two differently-scaled cross-sectional legs** (the `FundamentalQualityRanker` of ADR 0019 — `0.5·z_level + 0.5·z_trend`, each z-scored over the day's firing cohort), `StockRanker` gains a second, optional ranking mode: `rankCohort(candidates, context)` scores a whole same-day cohort at once, alongside the existing per-stock `score(stock, quote, context)`. The default `rankCohort` delegates to per-stock `score`, so every existing ranker is **byte-identical**; only a ranker that overrides it sees the cohort. It is wired into the engine at the **single call site where the full same-day cohort is co-resident** — the day-by-day selection loop in `BacktestService` (`resolvedEntries` for `currentDate`), immediately before tie-break jitter and the capital-aware sort.

## Why

- **A single-leg cross-sectional z is a no-op; a *blend* of two is not.** A z-score `(x − mean)/stdev` uses day-constant `mean`/`stdev`, so ranking by `z_level` alone is a *rank-preserving transform* of raw GP/TA — identical ordering, no cohort needed (the [rank-preserving-transform check](../../CONTEXT.md) every new ranker owes). The cohort z earns its place *only* because the ranker combines GP/TA (a ratio ~0.1–0.5) with an operating-margin YoY delta (a different scale): without standardizing each leg to the cohort, whichever leg has the larger raw variance silently dominates the 0.5/0.5 blend. Any blend of two cross-sectional legs (z-scores, or rank-averaging) requires the day's cohort; per-stock scoring structurally cannot express it.
- **The existing rankers don't already do this.** The residual-momentum rankers self-normalize within *one stock's own residual series* (`Σε / stdev(ε)` over that stock's estimation window) — comparability across stocks by construction, no cohort. They are not a cross-cohort normalizer to "copy"; the capability genuinely did not exist.
- **Why only one call site.** The engine scores in two places. Pass 1 (`collectEntrySignals`) scores per-stock across low-memory batches and stores `EntrySignal.rankerScore` — but that value feeds *only* the warmup-starvation diagnostic, which is skipped for any ranker with `warmupTradingDays() ≤ 0` (the quality ranker reads point-in-time fundamentals, not trailing bars, so its warmup is 0 and its Pass-1 score is computed-but-never-read). The day-by-day loop **recomputes** the score fresh against the co-resident `resolvedEntries` cohort and ignores the stored one. So the cohort is available exactly where selection happens, and nowhere else — wiring `rankCohort` there covers selection without disturbing the two-pass memory model (the same constraint that put the RS percentile in Midgaard, ADR 0009).

## Considered options

- **Add `rankCohort` with a per-stock default (chosen).** General — any future cross-sectional ranker (rank-blend, winsorized z, cohort-relative anything) reuses it. The default-delegation makes the change invisible to all existing rankers, verifiable by their unchanged trade lists.
- **Defer the ranker; ship the gate condition only.** Rejected by the operator: the cohort hook is reusable infrastructure worth building once, and ADR 0019 scopes the ranker into the same build. (The premise may still die at the `/condition-screen` kill-test, which the gate condition alone feeds — but the hook outlives this candidate.)
- **Self-normalize each leg to a fixed/pre-registered constant instead of the cohort.** Rejected — arbitrary constants are not what the quant specified (`n<2 → neutral` is explicitly cohort-sized) and would silently mis-weight the blend as the universe's profitability distribution drifts over 25 years.
- **Mutable `prepareCohort()` lifecycle hook that stashes cohort stats on the ranker instance.** Rejected — rankers are singletons from `RankerFactory`; per-instance mutable state is fragile across days/windows. `rankCohort` is a pure function of (cohort, context), no stashed state.

## Consequences

- `StockRanker` now has two ranking entry points. The contract: a ranker overrides **at most one** of `score` / `rankCohort`. `rankCohort`'s default = per-stock `score`; a cohort ranker's `score` is interface-required but unused by selection (it returns the neutral single-stock value).
- Degeneracy is defined per-leg per-day: a cohort of `n<2`, or a leg whose cross-sectional `stdev < 1e-12`, yields `z = 0.0` (neutral) for that leg; a name with the leg's input undefined (e.g. <8 quarterly filings for the margin-trend leg) is **excluded from that leg's mean/stdev**, then assigned `z = 0.0` — never `−MAX`, which would deterministically sink under-reported names.
- Tie-break jitter and the capital-aware sort are unchanged: `rankCohort` returns scores, the loop adds the same `TIE_BREAK_JITTER` and sorts as before, so a per-stock ranker's selection is bit-for-bit what it was.
- The cohort is "stocks passing the **whole** entry strategy on the day" (`resolvedEntries`), not "gate-passers" — the z-scores standardize over the actual competing set, consistent with the ranker's job of ordering same-day entry candidates.
